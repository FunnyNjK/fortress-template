# Deployment

Last Updated: 2026-05-08

## Deployment Status
Fortress Template deployment plan documented. No infra has been applied yet.
Phase 6 (Terraform) is where infra code gets written; actual Azure/AWS resources
are provisioned outside the AI workflow (one-time human setup).

---

## Local Development

1. Run `pnpm setup` (or `./scripts/setup.{sh,ps1}`) which:
   - Generates `.env` files for each app from `.env.example` with strong randoms.
   - Runs `docker-compose up -d` (Postgres, Redis, mailpit, Azurite, Unleash).
   - Runs Drizzle migrations.
   - Seeds a dev user.
   - Prints next steps.
2. Run `pnpm dev` — Turbo orchestrates `apps/web`, `apps/api`, `apps/worker`,
   and `apps/marketing` in parallel with hot reload.

Supporting services and their local ports:

| Service | Image | Port |
|---|---|---|
| Postgres 18 | `postgres:18` | 5432 |
| Redis 8 | `redis:8` | 6379 |
| mailpit (SMTP testing) | `axllent/mailpit` | 1025 (SMTP), 8025 (UI) |
| Azurite (Blob testing) | `mcr.microsoft.com/azure-storage/azurite` | 10000 (Blob) |
| Unleash (feature flags) | `unleashorg/unleash-server` | 4242 |

All apps run as native Node processes on Ubuntu — never inside containers.

## Production Environment

Same shape as staging in a separate Azure subscription. Requires manual approval
gate. Post-deploy smoke tests run automatically; automated rollback on health
check failure via Azure Container Apps revision management.

## CI/CD (GitHub Actions)

Triggers: `push` and `pull_request` (never `workflow_dispatch`-only).

### `.github/workflows/ci.yml` (every PR and push)
Jobs (in dependency order):
1. `install` — pnpm install with lockfile cache
2. `lint` — ESLint 10 type-checked rules
3. `typecheck` — `tsc --noEmit` per workspace
4. `test` — Vitest unit + integration (--run)
5. `e2e` — Playwright smoke (web)
6. `dep-audit` — `pnpm audit --audit-level=high` (blocking on high/critical)
7. `gitleaks` — secret scan
8. `semgrep` — SAST
9. `codeql` — SAST (separate workflow `codeql.yml`)
10. `sbom` — CycloneDX, uploaded as artifact
11. `docker-build` — build all images, no push

### `.github/workflows/deploy-staging.yml` (merge to `main`)
- All CI jobs pass → migrate (Drizzle, pre-deploy) → deploy to staging →
  smoke test → notify.

### `.github/workflows/deploy-prod.yml` (tagged release)
- Manual approval gate → migrate → deploy to prod → smoke test → automated
  rollback on health-check failure → notify.

## Secrets and Configuration

- **Local:** `.env` files generated by `scripts/setup.{sh,ps1}` with strong
  randoms. `.env` files are gitignored. `.env.example` is the source of truth.
- **CI:** GitHub Actions encrypted secrets (set per-repo, not committed).
- **Prod:** Azure Key Vault (default) / AWS Secrets Manager (AWS module).
- **Startup validation:** startup rejects `replace-with-*` placeholder values
  in `NODE_ENV=production`.

## Migrations

- Drizzle migrations are forward-only by default. Reversible migrations are
  explicitly flagged in the migration file header.
- Migrations run in a pre-deploy job, **not** at app startup.
- Dev: `pnpm --filter @fortress/api db:migrate` (runs against docker-compose Postgres).
- Prod: CI job `migrate` in `deploy-staging.yml` / `deploy-prod.yml`.

---

## Rollback

- **Azure Container Apps:** revision rollback via `az containerapp revision activate`.
- **AWS ECS:** rolling update reversion via ECS service update.
- **Database:** migrations are forward-only; data rollback follows
  `docs/runbooks/disaster-recovery.md`.

## Initial Azure Setup (one-time, outside AI workflow)

1. Create Azure Resource Group, VNet, private subnets.
2. Create Managed Identities (one per app, least privilege).
3. Create Azure AD app registration for GitHub deploy.
4. Configure federated credential on the app registration:
   - Issuer: `https://token.actions.githubusercontent.com`
   - Subject: `repo:<owner>/<repo>:ref:refs/heads/main`
   - Audience: `api://AzureADTokenExchange`
5. Create Container Apps Environment, Azure DB for PostgreSQL Flexible Server,
   Azure Cache for Redis (Premium), Azure Blob Storage, Azure Key Vault.
6. Add repo variables in GitHub:
   - `AZURE_CLIENT_ID`, `AZURE_TENANT_ID`, `AZURE_SUBSCRIPTION_ID`
7. Configure runtime secrets in Azure Key Vault (not in GitHub secrets).
